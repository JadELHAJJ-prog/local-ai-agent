## What it is

**ReAct** (Reason + Act) is the pattern that makes LLM agents useful. Instead of answering in one shot, the agent:

1. **Reasons** about what it needs
2. **Acts** by calling a tool
3. Sees the tool result
4. Reasons again with that new information
5. Either acts again or produces a final answer

This loop repeats until the agent has enough information. LangGraph makes this loop explicit as a graph cycle.

## The tool execution flow

Here's what happens when the user asks "what's the latest news about AI?":

```
1. agent_node called with [HumanMessage("what's the latest news about AI?")]
   -> LLM decides to call search_web
   -> Returns AIMessage(tool_calls=[{name: "search_web", args: {query: "latest AI news"}}])

2. should_use_tool sees tool_calls -> routes to tool_node

3. tool_node executes search_web("latest AI news")
   -> Returns ToolMessage(content="Title: AI breakthrough... URL: ...")

4. tool_node -> fixed edge -> agent_node (again)
   with [HumanMessage, AIMessage(tool_call), ToolMessage(result)]
   -> LLM reads the search results and writes a summary
   -> Returns AIMessage(content="Here are the latest AI developments...")

5. should_use_tool sees no tool_calls -> routes to output_parser_node

6. output_parser_node validates response -> routes to END
```

The graph cycle is: `agent_node -> tool_node -> agent_node`. The cycle breaks when the agent produces a response with no tool calls.

## ToolNode - the prebuilt executor

```python
# src/graph.py
from langgraph.prebuilt import ToolNode

graph.add_node("tool_node", ToolNode(tools))
```

`ToolNode(tools)` is a prebuilt LangGraph node that:
1. Reads the last AIMessage in state
2. Finds the tool_calls list
3. Looks up each tool by name in the tools list
4. Calls the tool function with the provided args
5. Wraps the result in a ToolMessage
6. Adds the ToolMessage to state

You don't write this code - LangGraph does it. The only requirement: the last message in state must be an AIMessage with tool_calls.

## The should_use_tool routing function

```python
# src/nodes.py
def should_use_tool(state: AgentState) -> str:
    last_message = state["messages"][-1]
    if last_message.tool_calls:
        tool_name = last_message.tool_calls[0]["name"]
        if tool_name == "execute_code":
            if not state.get("code_generated", False):
                return "code_generation_node"
            return "human_approval_node"
        return "tool_node"
    return "output_parser_node"
```

The execute_code path is special: code must be generated by the coder LLM and approved by the human before ToolNode runs it. All other tools go directly to tool_node.

## All 5 tools and their implementations

**search_web** - DuckDuckGo, no API key:
```python
@tool
def search_web(query: str) -> str:
    """Search the web for current information..."""
    with DDGS() as ddgs:
        results = ddgs.text(query, max_results=3)
        return "\n".join([f"Title: {r['title']}\nURL: {r['href']}\nSummary: {r['body']}" for r in results])
```

**execute_code** - Docker sandbox (see docker-sandbox.md for details):
```python
@tool
def execute_code(code: str) -> str:
    """Execute Python code safely in an isolated Docker container..."""
    # writes code to temp file, mounts into Docker, captures stdout/stderr
```

**analyze_image** - base64 encode and send to VLM:
```python
@tool
def analyze_image(image_path: str, question: str = "What is in this image?") -> str:
    """Analyze an image using vision AI..."""
    with open(image_path, "rb") as f:
        base64_image = base64.b64encode(f.read()).decode("utf-8")
    message = HumanMessage(content=[
        {"type": "text", "text": question},
        {"type": "image_url", "image_url": {"url": f"data:image/jpeg;base64,{base64_image}"}},
    ])
    response = vlm.invoke([message])
    return response.content
```

**analyze_video** - OpenCV frame sampling + batched VLM:
```python
@tool
def analyze_video(video_path: str, question: str = "What is happening in this video?") -> str:
    """Analyze a video using vision AI by sampling key frames..."""
    cap = cv2.VideoCapture(video_path)
    # samples MAX_FRAMES evenly spaced frames, encodes as base64, sends all to VLM in one call
```

**analyze_document** - text extraction + LLM analysis:
```python
@tool
def analyze_document(file_path: str, question: str = "Summarize this document") -> str:
    """Analyze a document file (PDF, Word, Excel, CSV)..."""
    # uses pdfplumber for PDF, python-docx for docx, openpyxl for xlsx, pandas for CSV
    # truncates to 6000 chars if too long, sends extracted text to LLM
```

## Gotchas and lessons learned

- **ToolNode requires AIMessage first.** The crash described in input-router.md happened because the router sent a HumanMessage directly to ToolNode. ToolNode can't work with a HumanMessage - it needs the preceding AIMessage that contains the tool_call specification.
- **Video analysis is slow.** Sampling 8 frames from a video, encoding them as base64, and sending a single large message to the VLM takes 20-60 seconds depending on video length. It's the correct approach (avoids multiple VLM calls) but the latency is real. I documented it as a known limitation.
- **Document truncation at 6000 chars.** Long PDFs are truncated before being sent to the LLM. This means the LLM might not see content from the end of a long document. A proper solution would chunk the document and process chunks separately. Left as a v2 improvement.
- **Ollama model swap latency.** When the agent calls `analyze_image` or `analyze_video`, Ollama needs to swap from the LLM to the VLM. This takes 3-5 seconds. The user sees the agent pause before the visual analysis begins.
