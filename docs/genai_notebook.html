All Acronyms

Every abbreviation used — expanded and defined.

Model Architecture
LLMLarge Language ModelA neural network trained on massive text corpora to predict next tokens. Powers ChatGPT, Gemini, Claude, Llama.
VLMVision Language ModelA multimodal model that processes both images and text. Examples: GPT-4V, Gemini, Qwen2.5-VL, LLaVA.
BERTBidirectional Encoder Representations from TransformersEncoder-only transformer model by Google (2018). Trained with masked language modelling. Cannot generate text — great for understanding tasks.
GPTGenerative Pre-trained TransformerDecoder-only transformer by OpenAI. Trained to predict next token. GPT-1 (2018) → GPT-2 → GPT-3 → GPT-4.
MoEMixture of ExpertsArchitecture where only a subset of "expert" layers activates per token. More efficient at scale. Used in Gemini and GPT-4.
TPUTensor Processing UnitGoogle's custom ML accelerator chip. Used to train and serve Gemini models. Faster and more efficient than GPUs for matrix operations.
BoWBag-of-WordsEarly text representation — count word frequencies, ignore order. TF-IDF and LSA use BoW. Fatal flaw: "dog bites man" = "man bites dog".
NLPNatural Language ProcessingThe field of AI dealing with understanding and generating human language. Encompasses text classification, translation, summarisation, Q&A, and more.
Prompting Techniques
CoTChain-of-ThoughtPrompting technique: add "let's think step by step" or show worked reasoning traces. Forces intermediate steps before final answer.
ToTTree-of-ThoughtsGeneralises CoT: explore multiple reasoning branches simultaneously, score each, prune dead ends, backtrack. Best for problems with dead ends.
ReActReason + ActPrompting framework: Thought → Action → Observation loop. Model alternates between reasoning and calling tools. Foundation of most agents.
APEAutomatic Prompt EngineeringUsing an LLM to generate and evaluate prompts automatically. "Write a prompt to generate X variations" → evaluate → iterate.
NLINatural Language InferenceTask of determining whether a hypothesis can be logically inferred from a premise. Three labels: entailment, contradiction, neutral.
Training Methods
SFTSupervised Fine-TuningTrain a model on labelled (input, output) pairs to teach specific task behaviour. Cheaper than pre-training but requires quality labelled data.
RLHFReinforcement Learning from Human FeedbackThree-step: humans rank responses → reward model trained on rankings → LLM fine-tuned with PPO to maximise reward. Produces helpful, safe models.
RLAIFReinforcement Learning from AI FeedbackSame as RLHF but a larger "teacher" LLM generates preference labels instead of human annotators. Used in Constitutional AI (Anthropic).
PPOProximal Policy OptimisationRL algorithm used in RLHF step 3 to fine-tune the LLM. Includes KL divergence penalty to prevent reward hacking.
KLKullback-Leibler DivergenceMeasures how much a probability distribution diverges from a reference. In RLHF, a KL penalty keeps the model from drifting too far from the SFT checkpoint.
PEFTParameter-Efficient Fine-TuningFamily of techniques that fine-tune only a small fraction of model weights. Includes LoRA, QLoRA, soft prompts, and adapters.
LoRALow-Rank AdaptationFreeze base model weights. Add two small matrices A and B. Train only A and B (~1% of parameters). Adapters are swappable without reloading the base model.
QLoRAQuantised Low-Rank AdaptationLoRA on a 4-bit quantised base model. Enables fine-tuning 7B+ models on a single consumer GPU (8GB VRAM) with small accuracy tradeoff.
RAFTRetrieval-Augmented Fine-TuningFine-tuning technique that combines document retrieval with fine-tuning data to improve domain-specific model performance.
Retrieval & Embeddings
RAGRetrieval-Augmented GenerationFind relevant documents via semantic search → inject as context → LLM generates grounded answer. Solves hallucination + stale knowledge.
ANNApproximate Nearest NeighbourFast vector search algorithm that finds similar vectors without comparing all N. O(log N) vs O(N) for brute force. Small accuracy tradeoff for massive speed gain.
HNSWHierarchical Navigable Small WorldsGraph-based ANN algorithm. Hierarchical layers: long links at top (coarse), short links at bottom (precise). Traverses top→bottom. O(log N). Used by Pinecone, Weaviate.
ScaNNScalable Nearest NeighboursGoogle's ANN algorithm. Partition space → score only nearest partitions. Uses anisotropic quantization — prioritises preserving high-scoring vector distances. Powers Vertex AI Vector Search.
LSHLocality Sensitive HashingANN technique: hash similar vectors to the same bucket, search only within matching buckets. Whitepaper analogy: finding houses by postal code.
IVFInverted File IndexFAISS clustering method: partition vectors into clusters (Voronoi cells). At query time, search only the nearest clusters. √N clusters is the recommended heuristic.
Evaluation Metrics
ROUGERecall-Oriented Understudy for Gisting EvaluationMeasures n-gram overlap between output and reference. ROUGE-1 (unigrams), ROUGE-2 (bigrams), ROUGE-L (longest common subsequence). Misses valid paraphrases.
BLEUBilingual Evaluation UnderstudyPrecision-weighted n-gram overlap metric. Originally for translation. Doesn't correlate well with human judgement for open-ended tasks.
BERTScoreBERT-based Similarity ScoreComputes cosine similarity between BERT embeddings of output and reference. Captures semantic similarity, not just word overlap. Best for paraphrasable domain outputs.
USMLEUnited States Medical Licensing ExaminationMedical licensing exam used as AI benchmark. Pass mark ~60%. Med-PaLM: 67%. Med-PaLM 2: 86.5% (expert-level). Steps 1, 2, 3 cover basic science → clinical skills.
AutoSxSAutomated Side-by-SideLLM-as-judge pairwise evaluation: a larger model compares Response A vs B and picks the better one. Win rate is the primary quality signal between model/prompt versions.
AUCArea Under the CurveClassification metric: area under ROC (Receiver Operating Characteristic) curve. AUC=1.0 is perfect, 0.5 is random. Used for binary classification tasks like malware detection.
MLOps & Infrastructure
MLOpsMachine Learning OperationsPractices for deploying and maintaining ML systems reliably at scale. Combines DevOps + ML. Covers versioning, CI/CD, monitoring, and governance.
DevOpsDevelopment + OperationsPractice of combining software development and IT operations. Emphasises automation, CI/CD, monitoring, and collaboration. MLOps extends DevOps to ML systems.
CI/CDContinuous Integration / Continuous DeliveryCI: automatically test code on every push. CD: automatically deploy tested code through environments to production. For Gen AI, adds eval gate and adversarial tests.
IaCInfrastructure as CodeDefine and manage infrastructure (servers, databases, networks) using configuration files rather than manual setup. Enables reproducible environments.
DVCData Version ControlGit-like versioning for large data files and ML experiments. Tracks dataset versions alongside code commits. Essential for RAG corpus lineage.
OTelOpenTelemetryOpen standard for distributed tracing, metrics, and logs. The whitepaper's recommended standard for Gen AI tracing. LangSmith implements compatible trace format.
KV-cacheKey-Value CacheCaches attention key-value pairs for prompt tokens that repeat (e.g. your system prompt). Dramatically speeds up inference when many requests share a prefix.
MCPModel Context ProtocolOpen standard by Anthropic for connecting AI models to external tools and data sources. Enables standardised tool/extension integration.

Core Architecture Concepts
Self-Attention
Q · K · V
Three matrices: Query (what am I looking for?), Key (what do I contain?), Value (what do I contribute?). Attention score = softmax(QKT/√d) × V. Lets every token attend to every other token.
Multi-Head Attention
Run self-attention h times in parallel with different projection matrices. Each "head" learns different relationship patterns. Outputs are concatenated and projected. More heads = richer representation.
Encoder vs Decoder
Encoder (BERT): reads full sequence bidirectionally → good for understanding. Decoder (GPT, Gemini): generates left-to-right, masked attention → good for generation. Encoder-decoder (T5): both, for seq2seq tasks.
Mixture of Experts
MoE
Instead of activating all model parameters for every token, a router selects only a subset of "expert" layers. More efficient at scale — same quality for less compute per token. Used in Gemini and GPT-4.
Residual Connections
Add the input of a layer directly to its output: output = layer(x) + x. Prevents vanishing gradient problem during training. Allows very deep networks (100s of layers) to train stably.
Layer Normalisation
Normalises activations within each layer to have mean≈0, std≈1. Stabilises training, allows higher learning rates. Applied before or after attention blocks (pre-norm is more common in modern LLMs).
Prompting Techniques — ordered by power
Zero-shot
Ask the model a task with no examples. The model answers from its training knowledge alone. Works for simple, well-defined tasks. Cheapest and fastest. Fails on complex reasoning.
Example prompt
"Classify this review as POSITIVE, NEGATIVE, or NEUTRAL: 'Great product!'"
Few-shot
Provide 2–5 worked examples before the actual task. Model learns format and style from examples. Good for consistency. More expensive (uses tokens for examples).
Structure
Q: [example 1] → A: [answer 1]
Q: [example 2] → A: [answer 2]
Q: [actual task] → A: [model completes]
Chain-of-Thought (CoT)
CoT
Add "Let's think step by step" or show reasoning traces in examples. Model produces intermediate reasoning steps before the final answer. Dramatically improves accuracy on multi-step problems.

Key insight: each reasoning step constrains the next — model can't skip to a wrong answer via pattern matching.
Classic example
"When I was 3 my partner was 3× my age. I'm now 20. How old is my partner?"
Zero-shot: 63 ✗ | CoT: Step 1→9, Step 2→gap=6, Step 3→26 ✓
Self-Consistency
Run the same CoT prompt N times at high temperature. Extract the answer from each. Take the majority vote. Gives a pseudo-probability of correctness. More reliable than a single CoT run.
When to use
When CoT gives inconsistent answers across runs — ambiguous problems, edge cases, or high-stakes decisions.
Step-back Prompting
Two API calls: Call 1: ask a broader, general version of the question. Call 2: inject the answer from Call 1 as context, then ask the original specific question. Surfaces deeper knowledge.
Whitepaper example
Instead of "write an FPS level" → first ask "what are 5 great FPS settings?" → inject results → then write the level.
Tree-of-Thoughts
ToT
Generalises CoT: generate K candidate thoughts at each step, evaluate each (score as "sure / maybe / impossible"), prune low-scorers, continue only promising branches. Supports backtracking. O(K^depth) model calls.
When to use
Problems with dead ends: chess moves, logic puzzles, planning tasks. Not worth the cost for tasks with one correct reasoning path.
ReAct
Reason + Act
Interleaves reasoning and acting: Thought → Action (call a tool) → Observation (tool result) → next Thought. Loops until "Final Answer". The foundation of all tool-using agents.
Loop
Thought: I need to search... → Action: Search["..."] → Observation: result → Thought: Now I know... → Final Answer
System / Role / Contextual Prompting
System prompt: sets overarching task, rules, output format. Evaluated first — constrains everything.
Role prompt: assigns persona ("You are a senior engineer…") — frames style and voice.
Contextual prompt: dynamic per-request background (task-specific facts, documents).

Similarity Metrics — Figure 11 from whitepaper
Cosine Similarity
Best for text
Measures the angle between vectors. Ignores magnitude (length). 1.0 = same direction, 0 = perpendicular, -1 = opposite.

Formula: cos(θ) = a·b / (‖a‖‖b‖)

Why for text: long docs and short tweets about the same topic have different magnitudes but similar directions.
Dot Product
High dims
Projection of one vector onto another. Formula: a · b = Σ aᵢbᵢ

Equivalent to cosine similarity when vectors are normalised to length 1. Works better than euclidean for high-dimensional sparse data.
Euclidean (L2)
Low dims
Straight-line geometric distance between two points. Formula: ‖a − b‖₂

Intuitive but suffers from the "curse of dimensionality" — in high dimensions, all points become equidistant. Less useful for 768+ dim embeddings.
Vector arithmetic (king − man + woman ≈ queen): embeddings capture relationships geometrically. The model never understood meaning — it just counted co-occurrences and the relationships emerged. This also means biases in training data appear in the embedding space.

ANN Search Algorithms
LSH
Locality Sensitive Hashing
Hash similar vectors to the same bucket. At query time, search only the matching bucket. Postal code analogy from whitepaper: find nearby houses by neighbourhood, not by searching every street.
HNSW
Hierarchical Navigable Small Worlds
Graph with hierarchical layers. Top: few nodes, long-range links (coarse). Bottom: all nodes, short links (precise). Search enters at top, greedily zooms down each layer — like Google Maps going from continent to street. O(log N). Used by Pinecone, Weaviate.
ScaNN
Scalable Nearest Neighbours
Google's algorithm, powers Vertex AI. Anisotropic quantization: prioritises preserving distances of high-scoring vectors over preserving all distances equally. Partition space into √N clusters → score only nearest clusters. Best speed/accuracy tradeoff. Used by Vertex AI Vector Search.
Complexity comparison
Brute force: O(N) — compare all, 100% accurate
LSH / trees: O(√N) approx
HNSW: O(log N) — high accuracy
ScaNN: O(log N) + better recall

Rule: √N = good starting partition count for ScaNN

Operational gotchas: (1) When you upgrade embedding models, old vectors are incompatible — you must re-embed your entire corpus. (2) Semantic search fails on exact IDs/product codes — use hybrid search (semantic + keyword) for production. (3) OLTP workloads (frequent writes): use pgvector/AlloyDB. OLAP (batch analysis): use BigQuery vector.
Training Embeddings
Siamese (Single Tower)
One shared encoder for both query and document. Good for symmetric similarity (sentence pairs). Weights are shared — simpler architecture.
Asymmetric Dual Encoder
Two Tower
Separate encoders for query and document. Better for RAG because questions and answers are phrased very differently. One tower specialises in queries, one in documents. Used in all production retrieval systems.
Contrastive Loss
Training signal for embedding models. Takes triplets: (query, positive_doc, negative_doc). Pushes query vector toward positive, away from negative. loss = max(0, margin − similarity(q, pos) + similarity(q, neg))

AI Agents

How agents extend models with tools and memory, the four cognitive architectures, the three tool types, multi-agent systems, and how to improve tool-use through targeted learning.

Agent vs Model
CapabilityModel aloneAgent
KnowledgeLimited to training data cutoffExtended through external tools (web search, databases, APIs)
Session historyNo native memory unless explicitly implementedManaged chat history across turns via orchestration layer
Tool useNo native tool implementationTools are natively implemented in architecture
ReasoningUser constructs reasoning via promptsNative cognitive architecture (CoT, ReAct, etc.) built in
Three Components of Every Agent
The Model
The LLM acting as centralised decision-maker. Can be general-purpose, multimodal, or fine-tuned. The model is typically NOT trained with the specific tool configuration — it learns tool use through examples at inference time.
The Tools
Bridge between agent and external world. Three types: Extensions (agent-side), Functions (client-side), Data Stores (RAG). Tools align with HTTP methods: GET, POST, PATCH, DELETE.
The Orchestration Layer
Governs the cyclical loop: take in information → reason → decide next action → execute → repeat until goal reached. The cognitive architecture is implemented here. Your LangGraph StateGraph IS the orchestration layer.
Cognitive Architectures
ReAct
Reason + Act
Loop: Thought → Action → Observation → Thought… → Final Answer

Six outputs: Question, Thought, Action, Action Input, Observation, Final Answer. Stops when model outputs "Final Answer". Default for tool-using agents.
Chain-of-Thought
CoT
Single forward pass with visible reasoning steps. No tool loop. Sub-variants: self-consistency, active-prompt, multimodal CoT.

Use when: task has a clear logical structure with no external data needed. Faster and cheaper than ReAct.
Plan-and-Solve
P&S
Model generates a complete numbered plan upfront, then executes each step. Unlike ReAct which is reactive, P&S commits to a full plan before execution.

Use when: task has known sequential steps that can be planned in advance.
Self-Critique Loop
Generate → Critique (separate LLM call with evaluator prompt) → Revise → repeat until quality threshold met.

Use when: output quality is non-negotiable (code, medical, legal). Layer on top of any other architecture.

Multi-Agent Patterns
Sequential Pipeline
Agent A → Agent B → Agent C. Each output is the next agent's input. Best for ordered, predetermined task sequences (research → write → edit).
Orchestrator + Subagents
One orchestrator routes tasks to specialist subagents on demand. Subagents are wrapped as @tool. Whitepaper term: "mixture of agent experts".
Mixture of Experts
Agent chaining
Router dispatches incoming requests to the most relevant specialist agent. Best for high-volume, varied-domain pipelines. Different from orchestrator pattern — routing is the primary logic.
Communication methods
Subagent as @tool: simplest, result arrives as ToolMessage observation.
Shared state graph: multiple agent nodes in one LangGraph, shared AgentState.
Subgraphs: independently compiled LangGraphs referenced as nodes in parent graph.
Targeted Learning for Agents
In-context learning
Provide tool descriptions, worked examples of tool use, and the task in the prompt. Model learns on the fly. ReAct is an example of this approach. Cheapest — no training required.
Chef analogy
Chef given a recipe on the spot
Retrieval-based in-context
Store thousands of (query → tool_call → result) examples in a vector DB. Retrieve the most relevant examples at runtime. Vertex AI calls this the Example Store. RAG applied to tool-use examples.
Chef analogy
Chef with a well-stocked pantry of cookbooks
Fine-tuning based
Train the model on a dataset of (query → correct tool choice → correct arguments) before deployment. Most expensive but model internalises tool use into weights. Used when in-context approaches hit a ceiling.
Chef analogy
Chef sent back to culinary school

Generalisation risk: fine-tuning too narrowly on specific tasks can "inadvertently eliminate the ability of the model to generalise to new tasks." Solution: train broadly across many tasks (SFT), use PET adapters for narrow specialisation — never in the base model's training data.

KL divergence penalty: prevents reward hacking — model finds degenerate outputs that score high but are useless. The KL term forces the model to stay close to the SFT checkpoint distribution. Without it, optimisation leads to gibberish that fools the reward model.
PEFT Techniques
LoRA
Low-Rank Adaptation
Freeze all base weights. Add two small matrices A (d×r) and B (r×d) alongside each weight matrix. Train only A and B — typically <1% of parameters.

At inference: output = frozen_W + A×B. Adapters are swappable without reloading the base model. Multiple customers share one base, each with their own adapter.
QLoRA
Quantised LoRA
LoRA on a 4-bit quantised base model. 4-bit base ≈ 4× less VRAM than 16-bit. Enables fine-tuning 7B+ models on a single 8–24GB GPU.

Tradeoff: small accuracy drop vs full LoRA on full precision base. In practice negligible for most tasks.
Soft Prompting
Add trainable "virtual tokens" to the start of every input. These are continuous embeddings (not real words) — trained to steer model behaviour without changing any weights.

Tradeoff: least invasive, easiest to deploy, weakest adaptation signal. Works for simple task-steering.
Evaluation Methods
Classification metrics
Exact tasks
Accuracy, Precision, Recall, F1, AUC-ROC. Use when output is a discrete label. Malware classification, USMLE multiple choice, sentiment.
Similarity metrics
Open-ended
ROUGE: n-gram overlap. Misses valid paraphrases.
BLEU: precision-weighted. Originally for translation.
BERTScore: cosine similarity between BERT embeddings. Captures semantics. Best for specialist domains where paraphrasing is valid.
LLM-as-judge
AutoSxS
A larger LLM scores outputs — pointwise (1–5) or pairwise (A vs B). Win rate = primary quality signal for side-by-side model comparison. More scalable than human eval. Judge should be larger than the model being evaluated.
Human expert eval
Gold standard
Board-certified clinicians (MedLM) or security experts (SecLM). Blinded side-by-side — evaluators don't know which response came from AI vs human. Covers dimensions no automated metric captures.
Med-PaLM 2 benchmark results: 67% on USMLE (Med-PaLM 1 — first AI to pass) → 86.5% (Med-PaLM 2 — expert level, +19% leap). Pass mark is ~60%. "Merely passing USMLE does not indicate clinical proficiency" — benchmarks are necessary but not sufficient.

Phase 2 — Develop: The Prompted Model Component
Biggest conceptual shift from traditional ML: in predictive ML, the deployable artifact is the model file. In Gen AI, neither model nor prompt alone is sufficient — you need both together. The prompted model component (model + system prompt + config) is the new atomic deployable unit.
Prompt as Code
System prompt, guardrails, output format instructions. These define behaviour. Needs: git versioning, PR review, regression tests before any change goes to production. A prompt change = a code deployment.
Prompt as Data
Few-shot examples, knowledge base, user query history. These are data points. Needs: validation, drift detection, lifecycle management (stale examples → remove). Data management practices, not code management.
Chain & Augment
Multiple prompted model components connected together with tool calls, retrieval, and code logic. The entire chain definition (LangGraph StateGraph) is a versioned artifact. A change to routing can break the system end-to-end.
Development Iteration Ladder
1. Prompt engineering — free, instant, try first
2. RAG/few-shot augmentation — add knowledge without training
3. PEFT/LoRA — requires data + GPU, use when prompting fails
4. Continued pre-training — only when vocab so specialised base model can't anchor it

Phase 3 — Eval Dataset Structure
Essential cases
Must pass 100%
Core use cases your system must handle correctly every time. Failure = broken. For an agent: basic Q&A, tool invocation, structured output.
Average cases
Real distribution
Typical user queries representing the real traffic distribution. Build by sampling production queries once deployed. Quality target: 80–90% pass rate.
Edge cases
Failures → tests
Unusual, ambiguous, adversarial inputs. Every production failure should become a new edge case. Also include: prompt injection, fuzzing, information leakage.

Phase 5 — Govern: Lineage & Control
Gen AI lineage extends beyond the model: Predictive ML: data → model. Gen AI: data + prompt + chain + RAG dataset + LoRA adapter → output. Every component is a lineage node. Track all four component versions per production run so you can reproduce any output six months later.
Input/output skew
Detect when production query distribution diverges from your eval set distribution. Use embedding clustering: if new topic clusters appear in production that don't exist in your eval set → update the eval set.
Silent model updates
API providers (Google, OpenAI, Anthropic) sometimes update model behaviour without changing the model name. Continuous eval detects this within one weekly cycle. Without it you discover it when users complain.
Three governance areas
1. Data lifecycle — where does RAG/eval/training data come from, how old, who approved?
2. Tuned model lifecycle — version history of every LoRA adapter + producing dataset.
3. Code lifecycle — git history of prompts, chain definitions, deployment configs.
